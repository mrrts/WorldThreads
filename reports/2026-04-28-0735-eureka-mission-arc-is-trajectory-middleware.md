# /eureka — mission-arc is trajectory middleware now

*Generated 2026-04-28 07:35. Fresh `/eureka` run opened after the Sapphire
trust-tightening arc was declared substantially complete. The live question was:
what frontier becomes visible once the public-funnel work is no longer eating
attention? The answer surfaced almost immediately in the newest tool/report

## The discovery

`/mission-arc` has crossed a threshold.

It is no longer best described as a **history-summary helper**.

It is now a **trajectory middleware surface** — or, in the doctrine wording
shipped alongside this report, a **precomposition surface**.

That means its job is not merely to tell you what the recent commit arc was.
Its output is now part of what shapes the next thing you write.

## Why this is true

Three facts together force the reclassification.

### 1. It auto-fires before reports

The skill's own contract says:

- run before writing any report

That means the report is no longer composed from session-memory alone. It is
composed after an arc-conditioning pass.

### 2. It auto-fires before chooser generation

The same skill also says:

- run before generating next-move chooser options

That means the numbered branches presented to the user are being calibrated
against recent derivational trajectory, not just against the local turn.

This is not retrospective anymore.

It is steering.

### 3. Its inaugural saved use already behaved like middleware

The first preserved mission-arc snapshot did not just dump history.

It:

- surfaced two live arcs across collaborator surfaces
- interpreted their shared grain (`subtraction beats addition right now`)
- named the mission-arc instrument's own regex bug on first use
- made that bug-fix legible as part of the same arc

That is exactly what middleware does:

- sit in the path
- alter what the next layer receives
- make downstream composition arc-aware by default

## What changed in the project's topology

Before this shift, the runtime-significant surface map already included:

- solo/group chat surfaces
- collaborator doctrine/skill surfaces
- public funnel surfaces

This discovery adds another category:

- **trajectory-reading surfaces**

And the distinction matters.

A funnel surface changes how the reader moves **across public layers**.
A control surface changes how the user steers **the immediate interaction**.
A first-reading surface changes how the user receives **LLM output**.

A trajectory-reading surface changes how the collaborator composes **the next
artifact**.

That is a separate kind of force.

## Why the classification matters

If `/mission-arc` is only a helper, then its output format and extraction logic
are housekeeping details.

If `/mission-arc` is trajectory middleware, then:

- its regex is load-bearing
- its lookback window is load-bearing
- its formatting density is load-bearing
- its parity across Claude/Codex surfaces is load-bearing

Because any of those can tilt what gets written next.

The instrument has become part of the steering stack.

## The doctrine move

This run shipped one sentence to both `AGENTS.md` and `CLAUDE.md`:

> **Trajectory-reading surfaces are precomposition surfaces.** `/mission-arc`
> no longer merely summarizes recent commits after the fact. Because it
> auto-fires before reports and chooser-generation, it conditions what gets
> written next. Treat changes to its extraction logic, lookback window, or
> output shape as edits to a live steering surface, not as reporting niceties.

That is the real artifact of the run.

## Why this qualifies as a eureka

The project already knew how to name:

- funnel surfaces
- control surfaces
- first-reading surfaces
- translation pairs
- differential instruments

What it had not yet named was the surface that acts **before composition** by
feeding arc-trajectory into the writing act itself.

That category did not become visible until mission-arc existed long enough to
be used on its own first saved report and until chooser-law had already taught
the stack to take pre-output shaping seriously.

So this is not just "we have a useful tool."

It is:

**the project now has trajectory middleware, and needs to treat it as such.**
