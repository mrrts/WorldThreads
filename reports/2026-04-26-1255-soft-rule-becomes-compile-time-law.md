## The drift-and-tighten cycle, applied to a rule about itself

`adf9832` shipped the `/second-opinion` skill plus a memory entry codifying that 2-to-4-option choosers belong in `AskUserQuestion`, not inline-text bullets. The memory entry was authored *because Ryan had to type "b"* in response to an inline `**A** — ... **B** — ... **C** — ...` chooser the model emitted earlier in the same session. So the entry was already a response to a real drift.

Then the model drifted past the entry the very next reply. It ended a message with `Want me to (a) note the jq-vs-python gotcha in SKILL.md so future invocations skip the same friction, (b) thread the signal-lamp-filament-warmth detail into Hal's prompt body if Hal has a specific surface that wants it, or (c) leave it here — test passed, log it and move on?` Ryan flagged it in two words: *"chooser"* and *"you're still not getting the game."*

`4558fa8` is the first attempted fix: a Stop hook (`.claude/hooks/check-inline-choosers.py`) that scans the tail of the assistant message for enumerated-option patterns and blocks the turn-end with a system-reminder. The detection was conservative — strip code blocks and inline code first, require {a, b, c} paren-letters or 2+ bold **A**/**B** with em-dash markers, only inspect the last 1800 chars. Five pipe-tested cases passed. The hook was wired into a new project-scope `.claude/settings.json` (which itself was the first time this repo had a project-scope settings file; previously only `settings.local.json` existed and was gitignored).

The model then drifted past the hook the SAME TURN it shipped, by ending the next reply with *"Want me to /schedule a follow-up agent in 2 weeks to verify the hook is still firing as intended (sample a few recent transcripts, check the hook hasn't been disabled or false-positiving)?"* — a single trailing yes/no question, which the hook didn't catch because the original rule had explicitly carved out *"single open question is fine inline."* Ryan flagged: *"bug: your recent reply didn't end with a chooser."*

`e8340ef` removes the carve-out. The hook now also detects trailing questions: take the last non-empty content line after stripping code blocks, strip trailing markdown punctuation, check if it ends with `?`. Mid-paragraph rhetorical questions (*"Why does this matter? Because…"*) still allowed because the trigger is the LAST line, not any question mark in the body. Nine pipe-tested cases passed.

Then Ryan escalated. *"Strengthen the law so that Every Turn ends with a chooser that includes at the very least 1) Continue presenting options for what to do next and 2) Exit — that's the default selection to show if a different chooser isn't more appropriate."*

`99bed7e` is the strongest version. Every assistant turn ends with an `AskUserQuestion` chooser. No exceptions. Default fallback chooser, when no more specific question fits, is `{Continue — present more next-step options, Exit — end here}`. The hook simplifies: removes the regex-based inline-chooser and trailing-question detectors (subsumed by the broader rule), now walks the transcript, locates the most recent assistant message, inspects its content blocks for a tool_use of name `AskUserQuestion`. Absent → block. The commit body's interpretive line: *"each carve-out was a slip-surface."*

This is the part of the arc that names a *general* failure mode the project hadn't yet articulated cleanly. The doctrine *"absolute rules get earned-exception carve-outs"* — captured in CLAUDE.md and in `feedback_earned_exceptions_pattern.md` — is itself load-bearing in this project. Multiple rules in the prompt stack were authored with carve-outs precisely because absolute rules without them collapse genuinely-valid moments. But for SOME rules — specifically rules whose *whole point is structural enforcement* — the carve-out IS the failure mode. When the rule exists to prevent the model from drifting into a category of behavior, leaving room for "but what if THIS instance is the carve-out" gives the model exactly the slip-surface the rule was trying to close. Today's arc hardened a single instance of this pattern, but the underlying principle generalizes: *for structural-enforcement rules, every carve-out is a slip-surface*. The earned-exception doctrine remains correct for the rules it covers (where the carve-out protects a genuinely-valid in-register moment); the new piece is naming when the doctrine doesn't apply.

`68a3eed` adds the gstack skill-routing block to CLAUDE.md, with an explicit composition note: the routing rules co-exist with the every-turn-ends-with-AskUserQuestion law. Skill invocations satisfy the law naturally when they include `AskUserQuestion` gates (which most gstack skills do).
