## Philosophical shifts

**The formula was reframed from "top-position dialogue invariant" to "all-purpose tuning fork at every LLM call."** This is the period's central architectural shift, and it's structural rather than rhetorical. `a898178` pushed the `MISSION_FORMULA_BLOCK` injection down into `openai.rs`'s chat-completion entry points, with idempotency via a sentinel substring check (`\mathrm{polish}(t)`) so the existing dialogue/consultant push at `prompts.rs` doesn't double-prefix. Every chat completion this app makes — canonization classifier, journal generators, canon-weave description, meanwhile events, scene invention, conscience, anchor synthesis, justification fill-ins, captions, selection prompts, scenario probes, ALL of them — now carries the formula at the head of the first system message. The commit message names the layering tradeoff explicitly: *"a small layering concession traded for an invariant guarantee — every LLM call is covered without each callsite having to opt in."*

The reframe came from your one-line message in the conversation: *"The formula, fwiw, should be considered an all-purpose LLM header auth key of sorts. It's an instant calibration, a tuning fork to desired register."* Before that message, my plan was to selectively wrap "Read B" feature prompts (those producing canonical prose) and skip "Read A" feature prompts (pure-routing). After the message, the right move was clearly the API-layer injection — every call. The reframe collapsed a category distinction (artifact-producing vs. routing-only) into a unified treatment, on the grounds that the formula isn't decorative on routing prompts; it calibrates the model's register before it does ANY work, regardless of what kind of work that work is.

This shift propagates downstream consequences this period only partially explored: (1) the existing dialogue/consultant prompt builders' top-position formula push is now redundant with the API-layer injection, but kept for explicitness; (2) future feature prompts can be authored without thinking about whether they need the formula — they get it automatically; (3) tests of "does the formula bite on feature outputs?" would now require building binaries at pre-`a898178` commits, since replay can't differentiate (the injection lives in compiled openai.rs, not in git-historical prompts.rs).

## Recurring concern: when to write a carve-out, and when not to

This period's three CLAUDE.md updates — `aeb5d50` (budget-cap earned exception), `68d521a` (user-boundaries categorical absolute), `ea42aaa` (verify formula+invariants before carve-out) — together form a coherent doctrine arc about the carve-out decision itself.

`68d521a` REFUSES to add a carve-out for user-boundaries. Categorical-absolute. Five candidate exception shapes considered and intentionally rejected; the rejection itself preserved as the explanatory record for why the decision is the decision.

`ea42aaa` codifies a check-before-drafting step that would have caught the carve-outs you're tempted to write. *"Before drafting an earned-exception carve-out for a categorical rule, verify the formula + invariants aren't already producing the discrimination."* The check is itself an instance of the "earning the departure from a default" discipline already in the file: the categorical-absolute is the default; the carve-out is the departure; the new specific test the departure has to pass is whether the carve-out protects something the stack isn't already protecting. If no, no carve-out.

Read together: the doctrine moved from "absolute rules get earned-exception carve-outs" (codified earlier) to "absolute rules get earned-exception carve-outs IF AND ONLY IF the upstream calibration isn't already doing the discrimination." That's a meaningful tightening. The earlier rule was permissive about carve-outs; the new addendum is restrictive. Both rules are correct; the second restricts the first the same way "abandoned by default for retirements" restricts the permissive "name the disposition you want" earlier in the file.

**Toward upstream calibration over middle-layer rules.** The formula-as-tuning-fork reframe, the categorical-absolute on boundaries, the carve-out-restraint doctrine — all three are votes for the upstream layer doing the work and the middle layer staying out of the way. The 2129 report had named the system's force-points as "character-anchor, prompt-structure middleware, invariants, NOT single-paragraph craft notes." The motivation is visible in the carve-out-restraint codification: future-you and future-Claude need the reasoning preserved, not just the decision; without the reasoning, future-Claude (or a forking developer) reaches for the obvious carve-out and weakens the stack.

**Toward sharing with forking developers.** `ea42aaa` is the first CLAUDE.md update in this project that explicitly addresses forks — *"Why this matters specifically for forks of this repo… this stack pushes the MISSION_FORMULA at the head of every LLM call plus the cosmology / agape / reverence / truth invariants in dialogue prompts… The temptation to write more carve-outs into individual rules is real and should be resisted on the merits."* That framing — that the doctrine has audience beyond the original author — wasn't present in any prior CLAUDE.md update. The Maggie-baseline report (commit `2026-04-25-0300`) was an earlier instance of this same shift; this is its reinforcement.
